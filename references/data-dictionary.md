# Data Dictionary (DD0 Gate) — Reference Protocol

Gate DD0 requires documenting every dataset column or field before it is used in any analysis. Column names lie. A field called `score` could be a raw value, a percentile, a z-score, or a categorical encoding. The data dictionary eliminates this ambiguity.

---

## Purpose

Undocumented columns cause silent errors:
- A column labeled `age` might contain age at diagnosis, age at sample collection, or age in months.
- A column labeled `status` might be binary (0/1), categorical (A/B/C), or free text.
- A column labeled `value` is meaningless without context.

Gate DD0 forces the researcher to inspect and document every column before using it. This is a prerequisite for all downstream analysis.

---

## When to Run

**Trigger:** Before using any column or field from a dataset for the first time.

Specifically:
- When loading a new dataset for the first time
- When adding new columns through feature engineering
- When joining two datasets (document new columns introduced by the join)
- When receiving data from an external source or collaborator

DD0 does NOT need to be re-run for columns already documented, unless the data source has been updated.

---

## What to Document

For each column, record the following fields:

| Field | Required | Description |
|-------|----------|-------------|
| `column_name` | Yes | Exact name as it appears in the data |
| `dtype` | Yes | Data type: numeric (int/float), categorical, boolean, text, date, identifier |
| `example_values` | Yes | 3--5 representative values from the actual data |
| `semantic_meaning` | Yes | What this column actually represents (not what the name suggests) |
| `units` | If numeric | Measurement units (years, kg, percentage, z-score, etc.) |
| `source` | Yes | Where this column comes from (original dataset, derived, external join) |
| `missing_rate` | Yes | Fraction of missing values (0.0--1.0) |
| `known_caveats` | No | Any known issues, biases, or limitations |
| `verified_from` | Yes | How meaning was verified: documentation, source code, manual inspection, author communication |

---

## Column Names Lie — Verification Protocol

Do NOT trust column names at face value. Always verify semantics through at least one of these methods:

1. **Official documentation** — Check the dataset's README, data dictionary, or codebook. Record the source.
2. **Source code inspection** — If the data was generated by a pipeline, read the code that created the column.
3. **Value inspection** — Examine the distribution, range, and example values. Do they match what the name implies?
4. **Cross-reference** — Compare with metadata or other columns. Does `age` plus `birth_year` equal the expected calendar year?
5. **Author communication** — If all else fails, contact the data creator.

The `verified_from` field in the data dictionary records which method was used. "I assumed from the name" is NOT acceptable verification.

## Output Format

The data dictionary is written to `data-dictionary.md` in the project's `.vibe-science/` folder:

```markdown
# Data Dictionary

## Dataset: survey_responses_v2.csv

| Column | Type | Examples | Meaning | Units | Source | Missing | Verified From |
|--------|------|----------|---------|-------|--------|---------|---------------|
| resp_id | identifier | R001, R002, R003 | Unique respondent identifier | -- | Original dataset | 0.00 | Documentation |
| q1_score | float | 3.5, 4.0, 2.1 | Likert scale response to question 1 | 1-5 scale | Original dataset | 0.02 | Documentation |
| age_group | categorical | 18-25, 26-35, 36-45 | Age bracket at time of survey | -- | Original dataset | 0.01 | Documentation |
| derived_risk | float | 0.12, 0.87, 0.45 | Risk score computed from q1-q5 | probability (0-1) | Derived (see pipeline.py:L42) | 0.00 | Source code |
```

## Running the Gate

DD0 is primarily a manual gate — the researcher documents columns by inspecting the data and its documentation. However, the script can auto-generate a skeleton:

```bash
# Generate skeleton data dictionary from a dataset
python scripts/dd0_gate.py --data dataset.csv --output data-dictionary.md

# Validate an existing data dictionary for completeness
python scripts/dd0_gate.py --dict data-dictionary.md --data dataset.csv --validate
```

**Arguments:**

| Flag | Required | Description |
|------|----------|-------------|
| `--data` | Yes | Path to the dataset file |
| `--output` | No | Path for the generated skeleton (default: `data-dictionary.md`) |
| `--dict` | No | Path to an existing data dictionary to validate |
| `--validate` | No | Check that all columns in the data have dictionary entries |

**Validation checks:**
- Every column in the dataset has an entry in the dictionary
- No dictionary entries reference columns that do not exist in the data
- Required fields (dtype, example_values, semantic_meaning, source, verified_from) are not empty
- Missing rate is consistent with actual data

## Integration with DQ1

The data dictionary feeds directly into DQ1 (Post-Extraction) checks:

- **Distribution plausibility** uses the documented `dtype` and `units` to set expected ranges.
- **Missing value checks** compare actual missing rates against the documented `missing_rate`.
- **Metadata cross-check** uses the documented `source` to verify provenance.

If DD0 has not been completed for all columns, DQ1 cannot run. This dependency is enforced by the gate system.

---

## Common Mistakes

1. **Documenting after analysis** — DD0 must happen BEFORE using a column, not retroactively. Retroactive documentation tends to rationalize assumptions rather than verify them.
2. **Copy-pasting column names as meanings** — "age means age" is not documentation. What age? Measured when? In what units?
3. **Skipping derived columns** — Engineered features need documentation too. Record the derivation formula and source columns.
4. **One-time only** — If the dataset is updated, the data dictionary must be reviewed and updated to reflect any changes.

---

*This protocol is domain-agnostic. It applies to any tabular, structured, or semi-structured dataset.*
